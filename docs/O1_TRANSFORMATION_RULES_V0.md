# O1 Transformation Rules v0

This document defines the first transformation rules of Protocol O1.

Its purpose is not to provide a full formal language.

Its purpose is to compress repeated O1 behavior into a small set of reusable transformation rules.

At this stage, O1 has already shown recurring shifts across:

- time
- causality
- observer-centered statements
- benchmark evaluation
- structural gain analysis

The correct next step is therefore to identify the standard transformations that O1 repeatedly performs.

These rules are the first bridge between descriptive method and formal compression.

---

## Why transformation rules are needed

So far, O1 has been expressed through:

- protocol steps
- case analysis
- benchmark cases
- pattern tags
- structural gain criteria

This is already useful.

But it still leaves an important question open:

**What does O1 actually do to a statement?**

A transformation-rule layer provides a first answer.

It does not replace case analysis.

It extracts the recurring operations beneath case analysis.

---

## Rule format

Each transformation rule is expressed in four parts:

1. **Human default**  
   The ordinary observer-centered formulation pattern.

2. **O1 transformation**  
   The structural operation applied under observer suspension.

3. **De-centered output type**  
   The kind of structure that becomes explicit.

4. **Typical gain**  
   The main form of structural gain produced by the rule.

---

# Rule 1 - Absolute -> Relational

## Human default
A property is expressed as intrinsic, self-standing, or absolute.

Examples:
- the object is still
- the sun rises
- the past is fixed

## O1 transformation
Test whether the property depends on:
- frame
- relation
- configuration
- access condition
- reversibility structure

If yes, replace intrinsic phrasing with explicit relational structure.

## De-centered output type
- frame-relative property
- relational configuration
- access-dependent asymmetry

## Typical gain
- hidden dependence becomes explicit
- false absoluteness weakens
- structure becomes more transferable

---

# Rule 2 - Phenomenology -> Structure

## Human default
A lived experience is treated as primitive structure.

Examples:
- time flows
- the present is now
- I see a tree

## O1 transformation
Replace lived immediacy with explicit structural description in terms of:
- ordering
- transition
- configuration
- stabilization
- access indexing

## De-centered output type
- ordered state difference
- local access index
- stabilized configuration
- transition relation

## Typical gain
- phenomenology loses ontological privilege
- structure becomes explicit
- description becomes less observer-bound

---

# Rule 3 - Narrative Arrow -> Conditioned Dependency

## Human default
A readable directional story is treated as explanation.

Examples:
- A causes B
- the event happened because of X
- the market crashed because investors lost confidence

## O1 transformation
Replace direct arrow-language with dependency under conditions, including where relevant:
- enabling conditions
- thresholds
- multi-factor interaction
- propagation structure
- feedback structure

## De-centered output type
- conditioned dependency
- configuration-dependent emergence
- distributed causal structure

## Typical gain
- explanation is separated from narratability
- dependency structure becomes visible
- one-source compression weakens

---

# Rule 4 - Privileged Center -> Local Index

## Human default
A local center is treated as universal or ontologically privileged.

Examples:
- the present is now
- here is the real frame
- this viewpoint is the natural one

## O1 transformation
Replace privileged-center language with explicit local indexing.

## De-centered output type
- local descriptive index
- access position
- frame-relative center

## Typical gain
- local access is preserved without being universalized
- center loses metaphysical privilege
- comparison across frames becomes possible

---

# Rule 5 - Container Language -> Transition Language

## Human default
A process is described as occurring inside a vague container.

Examples:
- things unfold over time
- duration passes between changes
- events happen in time

## O1 transformation
Replace container metaphor with explicit transition organization in terms of:
- state difference
- transition sequence
- configuration change
- ordering relation
- metric separation

## De-centered output type
- state-transition structure
- configuration ordering
- relational duration

## Typical gain
- vague backdrop language weakens
- process becomes analyzable
- temporal description becomes structurally sharper

---

# Rule 6 - Readable Explanation -> Local Compression

## Human default
A statement is treated as structurally sufficient because it is readable, memorable, or intuitively strong.

Examples:
- one decision changed everything
- fear causes violence
- technology makes people lonely

## O1 transformation
Treat readability as possible compression rather than as sufficient structure.
Ask what was omitted to make the statement readable.

## De-centered output type
- compressed explanation unpacked into structure
- local summary distinguished from full dependency map

## Typical gain
- explanation stops being equated with simplicity
- omitted structure reappears
- slogans become analyzable

---

# Rule 7 - Given Object -> Stabilized Configuration

## Human default
An object is treated as already given and self-identical.

Examples:
- I see a tree
- the object is there
- the thing exists as itself

## O1 transformation
Test whether objecthood should instead be described as:
- segmentation
- stabilization
- persistence-pattern
- relational configuration
- classification result

## De-centered output type
- stabilized configuration
- persistence under classification
- non-primitive objecthood

## Typical gain
- primitive givenness weakens
- objecthood becomes analyzable
- observer/object separation becomes less naive

---

# Rule 8 - Sequence -> Partial or Local Order

## Human default
Processes are described as one-line sequence.

Examples:
- events happen in sequence
- first this, then that
- one step leads to the next

## O1 transformation
Test whether event organization is better described as:
- partial order
- branching order
- local ordering
- dependency ordering
- parallel structure

## De-centered output type
- partial order
- branching structure
- local sequence inside larger configuration

## Typical gain
- serial cognition loses default privilege
- more general temporal organization becomes available
- hidden non-linearity reappears

---

## Compact transformation table

| Rule | Human default | O1 transformation target | Output type |
|---|---|---|---|
| R1 | absolute property | relation / frame / access | relational configuration |
| R2 | lived immediacy | ordering / transition / stabilization | structural description |
| R3 | narrative cause | conditioned dependency | dependency configuration |
| R4 | privileged center | local index | indexed frame position |
| R5 | container metaphor | transition organization | state-transition structure |
| R6 | readable explanation | compression analysis | unpacked structure |
| R7 | given object | stabilization / persistence | configured objecthood |
| R8 | linear sequence | partial/local order | branching or local ordering |

---

## Rule composition

Some cases require more than one O1 rule.

Examples:

### "Time flows"
- R2: phenomenology -> structure
- R5: container language -> transition language

### "The present is now"
- R2: phenomenology -> structure
- R4: privileged center -> local index

### "A causes B"
- R3: narrative arrow -> conditioned dependency
- R6: readable explanation -> local compression

### "Events happen in sequence"
- R2: phenomenology -> structure
- R8: sequence -> partial or local order

This suggests that O1 may work not only through isolated rules, but through rule composition.

This is promising for future formalization.

---

## Constraint on use

A transformation rule should not be treated as automatically valid in every case.

The correct application of a rule must still satisfy the anti-inflation requirement:

**Does the transformation reveal explicit structure, or only produce more careful language?**

If the answer is unclear, the rule application is weak.

---

## What these rules do not yet provide

These rules are not yet:

- a scoring system
- a mathematical formalism
- a symbolic grammar
- a complete ontology
- an automatic parser

They are only the first reusable transformation layer extracted from O1 practice.

---

## Minimal conclusion

The transformation rules defined here are the first compact operational core of O1.

They show that O1 is not only a descriptive protocol, but also a structured set of observer-suspension transformations.

If these rules remain stable under broader application, they may support the next stage of the framework:

- rule composition
- stronger gain criteria
- compact notation
- minimal formalization

If not, they should remain a useful but provisional compression layer.
